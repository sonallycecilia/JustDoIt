# Auditoria de Segurança — JWT / auth-service

> **Data:** 2026-07-03
> **Escopo:** `services/auth-service` (emissão, validação, refresh e revogação de JWT) e validação nos serviços consumidores (task, schedule, notification).
> **Contexto arquitetural:** os 4 microsserviços compartilham o mesmo segredo HMAC (`JWT_SECRET`). Não há Redis no projeto — refresh tokens e revogação usam a tabela MySQL `refresh_token` via JPA.

## Achados

| Severidade | Arquivo:Linha | Vulnerabilidade | Impacto | Correção |
|---|---|---|---|---|
| **Crítica** | `docs/secrets.md:15` | Segredo JWT de produção commitado em texto claro no git (arquivo rastreado, commit `26d9ed4`); `docs/security.md:22` ainda expõe o prefixo do segredo default antigo no histórico | Qualquer pessoa com acesso ao repositório forja tokens de acesso válidos para **todos os 4 serviços** (mesmo segredo HMAC compartilhado) e assume qualquer conta sem senha | Remover o arquivo do git, expurgar o histórico (`git filter-repo`), **rotacionar o JWT_SECRET em produção imediatamente** e mover segredos para um cofre (env vars no VPS / secret manager) |
| **Média** | `AuthService.java:102-105` + `JwtAuthFilter.java:33` | Logout e exclusão de conta (`AuthService.java:113-120`) revogam apenas o refresh token; **não existe blacklist** (nem Redis nem outra — revogação usa tabela MySQL `refresh_token`). O access token segue válido até 15 min nos 4 serviços, inclusive após a conta ser deletada | Sessão "encerrada" continua utilizável por até 15 min em task/schedule/notification; token de conta excluída ainda acessa endpoints protegidos | Aceitar formalmente a janela de 15 min (documentando) ou implementar blacklist compartilhada (Redis com TTL = `exp` restante) consultada nos filtros de todos os serviços; incluir claim `jti` para permitir revogação individual |
| **Média** | `services/*/src/main/resources/application.yml` (`jwt.secret`) + `task/config/JwtUtil.java:16-21` | Mesmo segredo HMAC simétrico distribuído aos 4 serviços; tokens sem claims `iss`/`aud` | Comprometer qualquer serviço (ex.: notification) permite **emitir** tokens válidos, não só validá-los — o auth-service deixa de ser a única autoridade | Migrar para assinatura assimétrica (RS256/EdDSA): auth-service assina com chave privada, demais serviços validam com a pública (endpoint JWKS); adicionar e validar `iss`/`aud` |
| **Média** | `AuthController.java:37-40` + `AuthService.java:45-46,64-70` | Enumeração de usuários: `/auth/check-email` (permitAll, sem rate limit) responde `registered: true/false`, e `/auth/register` retorna "Email já cadastrado" | Atacante mapeia quais e-mails têm conta (base para credential stuffing e phishing dirigido) | Rate limiting por IP nos endpoints públicos (ex.: Bucket4j); avaliar resposta neutra no registro (fluxo de confirmação por e-mail) — a enumeração via check-email é decisão de UX consciente, mas precisa de throttling |
| **Baixa** | `AuthService.java:85-95` | Rotação de refresh token existe e o usado é invalidado, mas **reuso de token já rotacionado não dispara alarme**: falha silenciosa sem revogar as sessões do usuário | Refresh token roubado e usado pelo atacante antes do usuário legítimo passa despercebido; atacante mantém a cadeia de rotação indefinidamente | Ao receber refresh token inexistente/já usado de um usuário conhecido, revogar todas as sessões dele (detecção de reuso, padrão OAuth token-family) |
| **Baixa** | `AuthService.java:73-78` | Oráculo de timing no login: e-mail inexistente falha rápido (sem bcrypt); e-mail existente executa `passwordEncoder.matches` (~100ms) | Enumeração de contas por medição de tempo, mesmo com mensagem genérica | Executar `matches()` contra um hash dummy quando o usuário não existe |
| **Baixa** | `JwtAuthFilter.java:33` + `task/config/JwtUtil.java:23-30` | Claim `type: access` é emitida mas nunca validada nos filtros (auth e demais serviços) | Sem impacto explorável hoje (refresh tokens são opacos, não JWT), mas se alguém emitir outro tipo de JWT com o mesmo segredo no futuro, ele será aceito como access token | Validar `type == "access"` no filtro de todos os serviços (`require("type", "access")` no parser jjwt) |
| **Baixa** | `MxEmailVerifier.java:41-96` | `/auth/check-email` não autenticado dispara lookups DNS (MX/A) para domínio arbitrário controlado pelo chamador, sem rate limit | Vetor de abuso: amplificação de consultas DNS e consumo de threads/latência no caminho público | Cache de resultados por domínio + rate limiting; já existe timeout de 4s, o que limita o dano |
| **Baixa** | `AuthController.java:62,69,76,86` | Controller re-parseia o token do header (`authHeader.substring(7)`) em vez de usar o principal já autenticado no `SecurityContext` | Sem falha explorável (o filtro já validou o mesmo header), mas é frágil: um futuro endpoint fora do `authenticated()` que copie esse padrão confiaria no token sem validar assinatura | Injetar `Authentication`/`@AuthenticationPrincipal` e ler o `userId` do contexto, eliminando o segundo parse |
| **Baixa** | `RefreshTokenRepository.java` + `RefreshToken.java:35-36` | Refresh tokens expirados só são removidos se usados ou no login/logout do próprio usuário — não há limpeza periódica | Crescimento indefinido da tabela `refresh_token` (higiene, não exploração) | Job agendado (`@Scheduled`) com `deleteByExpiresAtBefore(now)` |

| **Informativa** | `JwtAuthFilter.java:33` + `WebSecurityConfig.java:42-45` | **Validação — OK**: assinatura + expiração verificadas pelo `parseSignedClaims` em filtro `OncePerRequestFilter`; `anyRequest().authenticated()` cobre toda rota não listada no permitAll, padrão replicado nos 4 serviços | — | — |
| **Informativa** | `AuthService.java:85-100,167-185` | **Refresh — OK**: token opaco de 256 bits (`SecureRandom`), armazenado só como hash SHA-256, rotação a cada uso (inclusive expirado), sessão única por usuário (login revoga anteriores) | — | Ver linha Baixa sobre detecção de reuso |
| **Informativa** | `AuthService.java:102-105` | **Revogação — parcialmente OK**: revogação server-side real do refresh token (delete no banco), não é só o client "esquecendo" o token | — | Access token é a lacuna — ver linha Média |

## Prioridade de ação

1. **Imediato:** rotacionar `JWT_SECRET` em produção e expurgar `docs/secrets.md` do histórico do git.
2. **Curto prazo:** rate limiting nos endpoints públicos; detecção de reuso de refresh token; validar claim `type`.
3. **Médio prazo:** decidir entre aceitar a janela de 15 min pós-logout ou implementar blacklist compartilhada; avaliar migração para assinatura assimétrica (RS256/JWKS).
